Pieni tiivistelmä:

- **Sudo without password**  
  - Sudo voidaan määrittää toimimaan ilman salasanaa muokkaamalla sudoers-tiedostoa  
  - Hyödyllinen automaatiossa, koska komennot eivät pysähdy salasanakyselyyn  
  - Käytetään usein palvelinten hallinnassa  
  Oma huomio: helpottaa käyttöä, mutta voi olla riski jos oikeudet menevät väärälle käyttäjälle  

- **xkcd 149: Sandwich**  
  - Sarjakuva, jossa sudoa käytetään väärin tavalliseen asiaan  
  - Idea on, että sudo antaa järjestelmänvalvojan oikeudet  
  - Korostaa sitä, ettei sudoa pitäisi käyttää turhaan  
  Oma ajatus: hyvä muistutus siitä, ettei sudo ratkaise kaikkea  

- **Passwordless sudo with Ansible**  
  - Ansiblella voidaan automatisoida sudo ilman salasanaa  
  - Helpottaa useiden koneiden hallintaa  
  - Sama idea kuin sudoers-muokkauksessa, mutta koodina  
  Kysymys: miten tätä kannattaisi rajata turvallisesti  

- **ansible-doc**  
  - Komento `ansible-doc` näyttää moduulien dokumentaation  
  - Esim. copy: content, dest, src  
  - apt: name, state, update_cache  
  Oma huomio: helpottaa paljon, ei tarvitse koko ajan etsiä ohjeita  

 a) Sudoless  

Tässä tehtävässä tein käyttäjän, joka voi käyttää sudoa ilman salasanaa. Käytin olemassa olevaa käyttäjää *user* ja lisäsin sille sudo-oikeudet ilman salasanakyselyä. Lisäksi varmistin, että SSH toimii ilman salasanaa, jotta Ansible toimii sujuvasti.

Tämä vaihe oli tärkeä, koska ilman tätä automaatio pysähtyisi salasanakyselyyn. Tässä piti olla tarkkana sudoers-tiedoston kanssa.

Käytetyt komennot
sudo usermod -aG sudo user
sudo visudo
ssh-copy-id user@localhost

b) Antero

Tässä loin käyttäjän Ansiblen avulla ja lisäsin sille SSH-avaimen. Käytin user ja authorized_key moduleja. Tässä käytin käyttäjää user.

Tein playbookin ja ajoin sen, minkä jälkeen pystyin kirjautumaan ilman salasanaa. Tämä oli kätevä, koska kaikki hoitui automaattisesti.

Käytetyt komennot
micro user.yml
ansible-playbook user.yml
ssh user@localhost

c) Package

Tässä asensin kaksi pakettia Ansiblen avulla apt-moduulilla. Määrittelin paketit ja niiden tilan.

Tämä oli aika suoraviivaista, ja hyvä esimerkki siitä, miten Ansible ei tee turhia muutoksia.

Käytetyt komennot
micro package.yml
ansible-playbook package.yml

d) File

Tässä loin tiedoston Ansiblen avulla ja määrittelin sille oikeudet, omistajan ja ryhmän. Käytin oikeuksia "0600", joka tarkoittaa -rw-------.

Tämä tarkoittaa, että vain omistaja saa lukea ja muokata tiedostoa.

Käytetyt komennot
micro file.yml
ansible-playbook file.yml

e) Jotain muuta

Kokeilin lisäksi service-moduulia, jolla voi hallita palveluita. Sen avulla voi esimerkiksi käynnistää tai pysäyttää palvelun tai varmistaa, että se on käynnissä.

Tämä vaikutti hyödylliseltä, koska palvelinten hallinnassa ei riitä pelkkä asennus, vaan palveluiden pitää myös toimia oikein.

Komennot: ansible-doc service
